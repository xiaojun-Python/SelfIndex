# 当前系统结构说明

这份文档是给“现在要重新理解项目的人”看的。

如果只看一句话：

SelfIndex 当前已经完成了一条最小记忆链路，但还处在新旧结构并存的过渡期。

## 先理解这三个概念

### 原始文档

原始文档是“系统保存下来的原件”。

比如：

- 一条聊天消息
- 一段笔记
- 一封邮件
- 一篇日记

当前在数据库里的表名是：`raw_documents`

### 记忆单元

记忆单元是“为了检索方便，从原始文档中切出来的小块”。

它不是原文的替代品，而是原文的可检索表示。

当前在数据库里的表名是：`memory_units`

### 回溯

回溯就是：

“当搜索命中一条记忆单元时，系统还能知道它来自哪条原始文档、原文的哪一段。”

这是目前这套架构里最重要的能力之一。

## 当前数据库里为什么会有两套结构

因为项目正在重构中。

旧结构：

- `conversations`
- `messages`
- `chunks`

新结构：

- `raw_documents`
- `memory_units`

现在之所以要同时保留，是因为：

- 旧 Web 页面还在跑
- 旧数据还在用
- 新链路已经开始接管核心能力

所以你可以把当前项目理解为：

“外层界面还是旧的多，底层骨架已经开始换新。”

## 现在一条数据是怎么流动的

### 导入时

```text
导出文件
-> parser 解析出会话和消息
-> 每条消息变成 raw_document
-> raw_document 被切成 memory_units
-> memory_units 写入 SQLite
-> memory_units 的向量写入 Chroma
```

### 搜索时

```text
搜索词
-> 转成查询向量
-> 去 Chroma 搜相似的 memory_units
-> 去 SQLite 拿 memory_unit 对应的 raw_document
-> 把结果和回溯信息返回
```

## 为什么“和旧版看起来差不多”却仍然值得做

因为这一步主要解决的是“系统能不能持续演进”，而不是“页面看起来有没有变化”。

如果没有这一步，后面这些能力都会越来越难做：

- 多数据源接入
- 记忆摘要
- 标签和实体
- 更可解释的排序
- 代理安全访问记忆

换句话说：

这一步不是为了让今天马上变得更炫，
而是为了让后面每一步都不再建立在越来越乱的旧结构上。

## 你现在最应该记住的现实状态

### 已经完成的

- 新的 Archive / Memory 分层已经存在
- 新的最小检索接口已经存在
- 旧数据可以自举到新记忆链路
- 旧 Web 还能继续用

### 还没有完成的

- Web 页面还没有完全围绕新模型重做
- 检索排序还比较简单
- 摘要/标签/实体还只是最小雏形
- 新旧兼容代码还没有彻底收敛

## 现在怎么看代码最不容易迷路

建议只抓下面几个文件：

- `app/__init__.py`：应用启动时到底做了什么
- `engine/bootstrap.py`：为什么旧数据也能工作
- `engine/database.py`：数据到底存在哪里
- `engine/memory.py`：原始文档如何变成记忆单元
- `engine/retriever.py`：搜索结果怎么来的
- `scripts/import_legacy_data.py`：导入流程怎么走

先理解这 6 个文件，项目的 80% 主干就已经在你脑子里了。
