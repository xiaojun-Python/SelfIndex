# SelfIndex 架构说明

## 核心目标

SelfIndex 当前不是一个“人格化 AI 产品”，而是一个个人记忆系统原型。

第一阶段最重要的事情只有一件：  
把“原始资料 -> 派生记忆 -> 检索回溯”这条链路做通，并且尽量保持轻量、透明、可重建。

## 三层结构

### 1. Archive Layer

作用：保存原始材料。

当前落地：

- SQLite 表：`raw_documents`

这层强调：

- 原文尽量保留
- 来源、时间、外部 ID 可记录
- 可重新处理
- 可追溯

### 2. Memory Layer

作用：把原始材料加工成更适合检索的中间表示。

当前落地：

- SQLite 表：`memory_units`
- Chroma 向量索引

这层当前只做了最小能力：

- chunk 切分
- 轻量摘要
- 向量化

后面更适合继续放进来的内容有：

- 标签
- 实体
- 关系
- 可解释的排序信号

### 3. Recall Layer

作用：在当前查询上下文里，找出值得返回的记忆单元。

当前落地：

- `engine/retriever.py`
- `GET /api/memory/search`
- `GET /api/memory/<memory_unit_id>`

这层现在还是最小版本，只做：

- 查询向量化
- 相似度搜索
- 返回简洁结果
- 保证可以回到原始文档

## 当前兼容层

项目还保留着旧版模型：

- `conversations`
- `messages`
- `chunks`

它们的作用是：

- 让旧 Web 页面还能继续工作
- 让旧数据不需要一次性全部废弃
- 给新架构争取平滑迁移时间

所以现在项目会同时有“两套数据模型”并存，这不是设计混乱，而是过渡期的刻意安排。

## 当前一次搜索的路径

```text
用户输入查询
-> retriever 把查询转成向量
-> Chroma 返回命中的 memory_unit
-> SQLite 查出 memory_unit 对应的 raw_document
-> 返回结果 + 回溯信息
```

## 为什么现在体验上和旧版差不多

因为这一步的主要工作是“重建地基”，不是“换新 UI”。

用户表面上看到的仍然是：

- 输入关键词
- 得到结果
- 点开查看

但系统内部已经发生了关键变化：

- 检索对象从“旧 chunk”开始过渡到“记忆单元”
- 原始文档和派生记忆开始分层
- 新检索结果具备回溯能力
- 后续多数据源接入有了统一容器

## 现在最重要的判断

项目目前已经不只是一个脚本集合，也还没有到很稳定的大系统。  
更准确地说，它正处在：

“原型工具”向“可持续演化的系统雏形”过渡的阶段。

这个阶段最容易让人觉得复杂度突然上升，这是正常现象。
